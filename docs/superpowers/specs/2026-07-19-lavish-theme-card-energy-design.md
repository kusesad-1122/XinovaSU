# XinovaSU 华丽主题卡片与注能开关设计

日期：2026-07-19
状态：设计已确认，实施计划已完成，等待执行选择
范围：仅管理器 `com.xinsu.moe` 的 Compose UI

## 1. 目标

本功能在现有透明主题地基上，为每一套内置主题提供与原图或配色人格相符的华丽卡片装饰，并让所有二态 UI 开关在开启时播放主题专属的“能量注入”动画。

完成时必须同时满足：

1. 21 套立绘主题和 11 套配色主题共 32 套，全部命中独立的装饰配方。
2. 不允许以同一套装饰只替换颜色来冒充独立适配。
3. 首页英雄卡、监控卡、功能卡、设置卡和列表卡均有装饰，不出现裸卡。
4. Material 与 Miuix 使用同一主题规格和动画状态，分别保留半透明玻璃与真模糊材质。
5. 所有二态开关均经过统一注能组件，包括 Material `Switch`、Miuix `SwitchPreference` 和模块列表中的直接开关。
6. 动画不延迟业务状态更新，不改变点击语义，不因主题或图片异常导致功能不可用。
7. 32 套主题映射、开关状态机、降级策略和源码入口均有可失败检查。
8. 结构覆盖必须达到 32/32；用户提出的“99%适配”作为视觉契合目标，必须经过 32 套预览矩阵和代表性真机检查，编译通过不能替代视觉验收。

## 2. 范围边界

### 2.1 包含

- 主题装饰数据模型与 32 套专属配方。
- Material `TonalCard` 与 Miuix `MiuixGlassCard` 的统一装饰宿主。
- 五种卡片角色的布局、装饰密度和安全区策略。
- 二态开关的统一注能状态机、主题粒子和最近卡片共振。
- 默认主题与自定义配色的稳定回退。
- 减少动画、省电、低性能设备和图片失败时的降级。
- 单元测试、源码守卫、预览矩阵和 CI 编译验证。

### 2.2 不包含

- 不修改内核、`xnsusd`、Root 功能行为或配置格式。
- 不修改 `kernel/Kbuild`、签名证书锚点、keystore、Secrets 或发布流程。
- 不重做复选框、单选框和分段选择器的完整注能动画；它们只消费主题颜色与轻量按压反馈。这里的“开关”特指表达启用/禁用状态的二态 Switch。
- 不为 32 套主题复制 32 份页面代码。
- 不新增持续运行的重型 Shader、逐卡实时图片分析或逐帧位图处理。

## 3. 现有基础与依赖

本设计依赖现有主题个性化 Phase A：

- `TonalCard.kt` 是 Material 卡片入口。
- `MiuixGlassCard.kt` 是 Miuix 卡片入口并负责现有玻璃模糊。
- `ThemeTemplate.kt` 的 `BuiltInThemes.all` 登记 21 套立绘主题和 11 套配色主题。
- `CuratedThemeCatalog.kt` 定义 18 套立绘 token；`BuiltInThemeCatalog.kt` 另外定义 `Sakura Street Walk`、`Moonlit Silver Blue`、`Sea Breeze Song`，21 套立绘 token 已完整登记。
- `ExpressiveSwitch.kt` 是 Material 公共开关入口。
- Miuix 页面仍有多个 `SwitchPreference`，模块页面还有一个直接 `Switch`；它们需要收敛到本地注能组件。

实施分支必须从包含 Phase A 的代码提交创建。若 Phase A 先合并到 `main`，则从最新 `main` 创建；否则从 Phase A 分支创建，不能在旧主题地基上平行复制组件。

## 4. 架构

### 4.1 数据层：`ThemeDecorationSpec`

新增一个不可变、渲染器无关的主题装饰规格。建议字段如下：

```kotlin
@Immutable
data class ThemeDecorationSpec(
    val themeId: String,
    val motif: OrnamentMotif,
    val frame: FrameRecipe,
    val iconPedestal: IconPedestalRecipe,
    val ambient: AmbientRecipe,
    val energy: EnergyRecipe,
    val layout: CardLayoutRecipe,
)
```

其中：

- `OrnamentMotif` 表达樱花、冰晶、墨线、月轨、荆棘、海浪等图形语汇。
- `FrameRecipe` 定义描边层数、切角、角饰、分隔纹样和高光方向。
- `IconPedestalRecipe` 定义图标底座形状、光环和激活状态。
- `AmbientRecipe` 定义静态背景纹理与低频环境运动。
- `EnergyRecipe` 定义开关轨道、粒子、爆发形态、方向和时序偏移。
- `CardLayoutRecipe` 定义标题轨道、状态徽章、内容内边距和装饰安全区。

`ThemeDecorationCatalog` 以 `ThemeTemplate.id` 为唯一键。32 套内置主题必须使用显式条目；默认主题和自定义配色使用单独的稳定回退函数，不能被统计为内置主题的“配方”。

### 4.2 应用层：统一解析

主题 Provider 根据当前 `ThemeTemplate.id` 解析一次 `ThemeDecorationSpec`，通过 CompositionLocal 提供给卡片与开关。解析顺序固定：

1. 命中 32 套显式内置配方。
2. 当前为默认 `None` 时使用 XinovaSU 默认能量纹章。
3. 当前为自定义颜色时，根据主色、次色、明暗和圆角生成确定性的参数化配方。
4. 未知或损坏 ID 使用默认纹章并记录非敏感诊断，不抛出 UI 异常。

自定义配方必须确定性生成：相同输入永远产生相同装饰，不使用运行时随机种子。

### 4.3 组件层：装饰卡片宿主

Material 与 Miuix 的公共卡片包装器内部增加同构装饰宿主：

```text
玻璃或半透明底层
  -> 主题边框与材质纹理
  -> 内容背后的环境装饰
  -> 原有交互内容
  -> 前景角饰与注能共振层
```

装饰层只绘制，不接收指针事件、不修改语义树、不持有业务状态。现有卡片调用即使不传新参数，也以 `Standard` 角色获得完整装饰，确保迁移过程中不会出现裸卡。

建议新增卡片角色：

```kotlin
enum class DecoratedCardRole {
    Hero,
    Monitor,
    Function,
    Standard,
    Compact,
}
```

页面仅选择角色和可选状态，不直接选择纹样。纹样始终由当前主题决定。

### 4.4 交互层：最近卡片共振

每个装饰卡片内部提供一个局部 `EnergyPulseHost`。卡片内的开关启动注能时，将进度发送给最近的宿主，使该卡片的边框和角饰共振；不会触发全屏所有卡片。

独立存在于对话框或无卡片容器中的开关仍播放自身轨道、光环和粒子，但不要求卡片共振。若同一卡片内多个开关连续操作，新的脉冲从当前进度接管并取消旧脉冲，禁止叠加无限动画任务。

## 5. 五类卡片布局

### 5.1 `Hero`

- 非对称标题轨道和最大面积主题纹章。
- Root/内核状态使用独立状态徽章，不与标题挤在同一行。
- 允许一处大角饰和一处前景光带；主体文字区必须有安全渐变。
- 注能时整张卡边框完成一次方向明确的共振。

### 5.2 `Monitor`

- CPU、温度、充电、电池和内存采用响应式仪表网格。
- 每项数据拥有主题化刻度、图标底座和小型能量槽。
- 数字保持最高层且不透明；装饰不得穿过数字基线。
- 2 秒数据刷新不得重新启动装饰或开关动画。

### 5.3 `Function`

- 功能图标、功能名、说明、启用状态和主开关形成清晰五段结构。
- 已选应用数量使用状态徽章展示。
- 开启后边框进入静态激活态；只有状态切换时播放爆发动画。

### 5.4 `Standard`（设置与普通内容）

- 分组标题使用主题轨道或丝带，不再只是普通文本上方留白。
- 开关右侧保留至少 48dp 触摸区域，装饰不能侵入。
- 长说明可换行，文字安全区优先于角饰完整度。

### 5.5 `Compact`

- 列表项只保留一处角标、一条侧边能量线和一处图标底座。
- 禁止常驻粒子覆盖长列表。
- 警告、授权、安装日志等高可读场景可提高底层不透明度，但仍保留主题边线与角标。

## 7. 注能开关状态机

### 7.1 状态真源

`checked` 始终是唯一业务状态真源。动画进度只是视觉状态：

- 用户点击时立即调用一次 `onCheckedChange`。
- 动画不能等待持久化、Root 服务或异步业务完成。
- 如果业务层随后把状态改回，动画从当前进度平滑反向。
- 页面首次进入且持久化状态已经为开时，不播放完整开启动画，只显示静态激活态。
- `enabled == false` 时不接受点击、不产生注能粒子。

### 7.2 开启时间轴

开启总时长目标为 900ms：

| 时间 | 阶段 | 视觉行为 |
|---|---|---|
| 0–120ms | 压缩 | 滑块轻微压缩，周围光线向触点聚集 |
| 120–300ms | 蓄能 | 主题能量在滑块内部形成核心 |
| 300–500ms | 灌注 | 能量穿过轨道，轨道纹样逐段点亮 |
| 500–720ms | 共振 | 光波扩散到最近卡片，边框与角饰响应一次 |
| 720–900ms | 沉降 | 主题粒子爆发并沉降为静态激活微光 |

每套主题可以改变粒子形态、路径、方向、旋转和阶段内偏移，但不能改变状态语义或把总时长扩展到影响操作节奏。

### 7.3 关闭与快速反转

关闭目标时长为 360ms：静态高光熄灭、粒子回流、轨道抽空、滑块恢复。快速连续切换时，动画从当前标准化进度反向，不重新从 0 或 1 开始，不并行启动第二套无限动画。

### 7.4 组件收敛

- Material：`ExpressiveSwitch` 委托给新的共享注能核心。
- Miuix：新增本地 `EnergyMiuixSwitchPreference`，复刻现有偏好行的信息层级和语义，但使用共享注能核心绘制开关。
- 模块页面：直接 `Switch` 改用共享核心。
- 对话框或无卡片宿主的开关：保留自身轨道与粒子，不执行卡片共振。

## 8. 性能预算与降级

### 8.1 绘制预算

- 使用 Compose Canvas、缓存 Path/Brush 和 `drawWithCache` 一类缓存机制。
- 不在 `draw` 热路径解析主题、分配集合、解码位图或执行图片取色。
- 英雄卡粒子 8–14 个，标准卡 4–8 个，紧凑卡 2–4 个。
- 华丽粒子只在状态切换时运行；常驻环境动画周期为 6–12 秒。
- 不新增动画模糊半径；Miuix 沿用现有玻璃模糊，Material 沿用半透明玻璃。
- 列表离屏项由现有 Lazy 容器释放后不得继续运行动画任务。

### 8.2 降级优先级

1. 系统关闭动画或要求减少动态效果：完整时间轴替换为约 120ms 的静态高亮过渡。
2. 系统省电模式：关闭常驻漂浮粒子，保留点击注能动画。
3. 玻璃卡片关闭：保留边框、角饰和注能，只提高底层不透明度。
4. 背景图缺失或解码失败：继续使用当前主题颜色和装饰，不回到白色裸卡。
5. 未知主题：使用 XinovaSU 默认纹章并保持全部功能可用。

### 8.3 性能验收

代表性真机在 60Hz 下播放单个标准卡注能时，采样区间的渲染帧时长 p95 目标不高于 16.7ms，且不能连续出现两个以上丢帧；滚动长列表时不应存在不可见卡片的持续粒子任务。该视觉性能项在实现完成前标记为 `[UNVERIFIED]`，必须通过真机性能测量后关闭。

## 9. 可访问性与安全区

- 正文对比度目标至少 4.5:1；大字、图标和边界至少 3:1。
- 文字和图标颜色不继承卡片透明度。
- 装饰不得成为表达开关状态的唯一方式；滑块位置、轨道形态和系统语义仍需区分开关状态。
- 开关触摸目标至少 48dp，装饰层不接收触摸。
- 大字体时优先扩展卡片高度或隐藏次要角饰，不能截断主要状态。
- 右到左布局下，能量方向跟随逻辑起止方向；主题图案中没有方向语义的纹样可以保持原构图。
- 减少动画模式下不播放旋转、爆发或快速闪烁。
- 任何故障闪烁只允许一次低频亮度变化，禁止高频频闪。

## 10. 错误处理

- 配方查找必须返回非空结果；未知 ID 返回默认规格。
- 可选纹样资源缺失时跳过该纹样并继续绘制边框，不能让卡片组合失败。
- 注能动画被取消、页面离开或状态反转时，协程与动画状态立即结束或接管，不泄漏到已释放页面。
- Miuix 本地偏好组件必须保留标题、摘要、启用状态、点击处理和 Switch 语义；不能为了动画丢失第三方组件原有行为。
- 动画组件不捕获或吞掉业务异常；业务回调正常返回但状态未更新时，开关视觉最终以外部 `checked` 为准。
- 所有诊断不得包含用户图片内容、文件字节、签名材料或其他敏感值。

## 11. 测试与验证

- 业务页面不得直接调用 Miuix `SwitchPreference`。

## 12. 分阶段实现与回滚点

### 阶段 1：规格模型与覆盖测试

新增数据模型、32 套目录和失败测试，不接入生产卡片。

通过条件：32/32 显式映射、唯一指纹和 21/21 立绘 token 完整性测试通过。

### 阶段 2：静态卡片装饰

实现共享装饰渲染器，接入 `TonalCard` 与 `MiuixGlassCard`，完成五类角色和默认回退，不加入开关动画。

通过条件：源码守卫证明业务卡片全部经过包装器；预览画廊能展示 32 套标准卡。

### 阶段 3：共享注能开关

实现状态机、Material 入口、Miuix 本地偏好组件、直接模块开关迁移和最近卡片共振。

通过条件：状态机错误路径测试通过；开关源码守卫无裸调用；双渲染器编译通过。

### 阶段 4：页面布局角色化

将首页英雄卡、硬件监控、功能、设置和列表逐步标记为正确角色，并调整内容结构与安全区。

通过条件：五类角色均在实际页面出现；长文字、大字体和紧凑列表不遮挡。

### 阶段 5：视觉与性能收口

完成 32 套预览矩阵、代表性真机检查、减少动画和省电模式验证。

通过条件：视觉主观验收通过；无离屏持续动画；CI `build-manager` 全绿。

每个阶段必须独立可回滚。若注能动画出现性能或兼容性问题，可以回退阶段 3，而保留阶段 1/2 的静态主题装饰；不得以删除 32 套配方来规避问题。

## 13. 完成定义

只有满足以下条件才能声称功能完成：

1. 32 套内置主题全部命中唯一专属配方。
2. 五类卡片都有装饰，Material/Miuix 无裸卡入口。
3. 所有二态 Switch 都使用共享注能组件，无裸 `SwitchPreference`。
4. 开启、关闭、快速反转、禁用和减少动画测试通过。
5. 默认、自定义、未知主题和图片失败均有稳定回退。
6. 源码守卫、单元测试、`git diff --check` 和 `build-manager` 均通过。
7. 32 套预览矩阵经人工确认，视觉项从 `[UNVERIFIED]` 转为通过。
8. `kernel/Kbuild` 签名锚点、内核、`xnsusd` 和发布配置保持未修改。
